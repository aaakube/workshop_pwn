## 1. The vulnerable program

```c
#include <errno.h>
#include <fcntl.h>
#include <stdio.h>
#include <stdlib.h>
#include <string.h>
#include <unistd.h>

int win(void)
{
    char flag[256];
    puts("You win! Here is your flag:");
    int fd = open("/flag", O_RDONLY);
    int n = read(fd, flag, sizeof(flag));
    write(0, flag, n);
    puts("\n");
}

int challenge(void)
{
    char input[111];
    puts("input:");
    int n = read(0, input, 4096);
    printf("Your entered: %s", input);
    puts("Goodbye!");
}

int main(void)
{
    puts("Good morning VA team :O");
    challenge();
}
```

**Can you spot the vulnerability?**

`input` is a local buffer of only 111 bytes, but `read()` is told it may write up to **4096** bytes into it. Nothing stops the caller from sending more than 111 bytes — the extra bytes spill past the end of `input` and overwrite whatever else lives on the stack right after it (saved registers, and eventually the return address). This mismatch between the buffer's real size and the size passed to `read()` is the classic stack buffer overflow.

A few instructions worth understanding on their own:

### `lea` — load effective address

`lea` (Load Effective Address) computes an address from its operand and stores *that address* in the destination register — it never dereferences memory. `lea rax, [rip+0xe35]` means: take the current value of `rip`, add `0xe35`, and put the resulting address into `rax`. This is exactly how the string literals ("input:", format strings, "Goodbye!") get referenced — as rip-relative addresses baked in at compile time.

### `read()`

`read(fd, buf, count)` tries to read up to `count` bytes from file descriptor `fd` into the buffer starting at `buf`. In `challenge()`, `fd = 0` (stdin), `buf` is `input`, and `count = 0x1000` (4096) — far larger than the 111-byte buffer it's writing into.

### `puts()`

`puts(s)` writes the string `s` followed by a trailing newline to stdout.

## 3. A process in memory

When a binary runs, the OS lays its address space out in distinct regions:

- **Text segment** — the compiled code, read + execute only
- **Data / BSS segments** — global and static variables
- **Heap** — grows upward, for dynamically allocated memory
- **Memory mapping segment** — shared libraries (e.g. `libc.so`) and other file mappings
- **Stack** — grows *downward*, holds local variables, saved registers, and return addresses
- **Kernel space** — off-limits to user code; touching it causes a segmentation fault

You can inspect this layout for a running process yourself:

```
cat /proc/<pid>/maps
```

which lists every mapped region (the binary itself, the heap, `libc`, the stack, `ld-linux`, etc.) along with its address range and permissions.

## 4. Key registers

Three registers matter most for understanding stack frames:

| Register | Role |
|---|---|
| `rbp` | Base pointer — a fixed reference point for the current function's stack frame |
| `rsp` | Stack pointer — points to the current "top" of the stack (the lowest used address) |
| `rip` | Instruction pointer — the address of the next instruction to execute |

In GDB you can see all of this with:

```
(gdb) info registers
```

and narrow it down with:

```
(gdb) info registers rip rsp rbp
```

## 5. Building a stack frame, step by step

At the start of `challenge()`:

```
push   rbp              ; save caller's base pointer
mov    rbp, rsp          ; establish new base pointer = current stack pointer
add    rsp, 0xffffffffffffff80   ; reserve local variable space
```

That third instruction looks alarming, but `0xffffffffffffff80` is just the 64-bit two's-complement encoding of **-128**. So `add rsp, -0x80` is functionally identical to `sub rsp, 0x80` — it reserves **128 bytes** on the stack for locals. The compiler emits it as an `add` of a negative constant here, but the effect is the same as a subtraction.

### A little bit of math

The size of a stack frame can be measured directly as the gap between `rbp` and `rsp`:

```
stack frame size = $rbp - $rsp
```

Using the register values captured right after the prologue:

```
rip   0x4011dc     0x4011dc <challenge+8>
rsp   0x7fffffffdf80
rbp   0x7fffffffe000
```

```
0x7fffffffe000 - 0x7fffffffdf80 = 0x80 = 128 (decimal)
```

This confirms it: 128 bytes reserved, even though the source only declared `char input[111]`. The extra bytes are padding — often there to keep `rsp` aligned to a 16-byte boundary before calls, per the x86-64 calling convention.

### What does the stack frame actually contain?

Dumping memory at `rsp` right before `read()` runs shows the buffer sitting at the bottom of the frame, filled with filler bytes (`0x61` = `'a'` in this example) up until the region just past `rbp`, where the **saved return address** appears:

```

```


This is the crux of the vulnerability: `input` sits *below* the saved return address on the stack. Overflow the buffer far enough, and you overwrite that return address — redirecting execution wherever you want once `challenge()` hits its `ret`.

## 6. Stack frames, the big picture

Every function call pushes a new frame on top of the stack. For a chain like `main()` → `bar()` → `truc()`:

```
Low addresses (stack grows this way ↑)
┌───────────────────────┐
│ Stack frame of truc    │  y, return address of truc
├───────────────────────┤
│ Stack frame of bar     │  y, x, return address of bar
├───────────────────────┤
│ Stack frame of main    │  result, return address of main
└───────────────────────┘
High addresses
```

Each frame holds that function's locals plus the address execution should return to once the function finishes. This is exactly the structure a stack overflow attacks: write past your own frame's locals, and you start clobbering the *next* frame up — most importantly, its saved return address.

## 7. Mitigations (for later labs)

This introductory pass deliberately ignores three protections you'll meet in later, harder labs:

- **ASLR** (Address Space Layout Randomization) — randomizes where the stack, heap, and libraries load each run
- **NX** (No-eXecute) — marks the stack non-executable, so you can't just jump into shellcode sitting on it
- **PIE** (Position Independent Executable) — randomizes where the binary's own code loads, not just libraries

For the first labs, ASLR is disabled, which keeps addresses predictable and lets you focus on the mechanics above before dealing with defeating these mitigations.

## 8. Recap

- The vulnerability: `read()` is told to accept up to 4096 bytes into a 111-byte buffer.
- `rbp` and `rsp` bound the current stack frame; `rip` always points to the next instruction.
- `add rsp, <negative number>` is just how a subtraction can be encoded — it reserves stack space, here 128 bytes for a 111-byte buffer plus padding.
- The saved return address lives just above the current frame, right past `rbp`.
- Overflowing the buffer far enough lets you overwrite that return address and redirect execution once the function returns.
